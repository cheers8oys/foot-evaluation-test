# Engineering Guardrails

This document expands the mandatory rules summarized in `AGENTS.md`.

## 1. Assumptions and Uncertainty

- State assumptions explicitly when they affect implementation shape, data contracts, or user-visible behavior.
- If multiple interpretations are plausible, present them instead of choosing one silently.
- If the simpler approach is sufficient, say so and prefer it.
- If something is unclear and cannot be derived from the repo or docs, stop and ask.
- Do not hide confusion behind implementation.

## 2. Simplicity First

- Write the minimum code that solves the task.
- Do not add features, flexibility, or configurability that were not requested.
- Do not introduce abstractions for one-off code.
- Do not add error handling for impossible states.
- If a solution feels large for the problem, simplify it before finalizing.
- Use the senior engineer test: if the change looks overbuilt, cut it down.

## 3. Surgical Changes

- Touch only files and lines required for the request.
- Do not refactor or restyle adjacent code unless the request requires it.
- Match local conventions even if you would choose a different style from scratch.
- Remove only the dead code created by your own change.
- If you notice unrelated issues, mention them separately and leave them untouched.

## 4. Goal-Driven Execution

- Define success as a concrete, checkable outcome before implementing.
- For validation work, add or update tests that prove invalid inputs fail.
- For bug fixes, add or update a test that reproduces the bug first.
- For refactors, preserve behavior and verify before and after.
- For multi-step tasks, keep a short plan with explicit verification at each step.

## 5. TDD and Verification

- Use TDD for product code when practical: failing test, minimal implementation, passing test, then cleanup.
- Domain logic, validation, API handlers, state utilities, and integration adapters need automated tests.
- UI changes should include render or interaction tests when practical. If not practical, record manual verification steps in the final summary.
- Code is not done until formatter, linter, typecheck, tests, and build pass.
- If tests are missing by necessity, state the gap and residual risk explicitly.

Required local checks:

```bash
npm run format
npm run lint
npm run typecheck
npm run test:run
npm run build
```

Pre-push check:

```bash
npm run verify
```

## 6. Security and Data Handling

- Never commit secrets or credentials: `.env`, `.env*.local`, API keys, access keys, sender keys, service account JSON, private keys, token secrets, or provider credentials.
- Never expose server-only secrets through `NEXT_PUBLIC_` variables.
- Never commit real personal data: names, phone numbers, CRM exports, API payloads with personal data, screenshots with personal data, or logs containing personal data.
- Never log full request or response bodies when they may include names, phone numbers, message content, result URLs, or provider credentials.
- Log only safe operational data such as internal error code, provider result code, and message group ID.
- Keep `.gitignore` aligned with local artifacts generated by tools.
- Do not run `npm audit fix --force` without explicit approval.

## 7. Git Discipline

- Start by checking `git status --short`.
- Stage only intentional files. Do not use `git add .`.
- Inspect staged content before commit:

```bash
git diff --cached --stat
git diff --cached
```

- Do not commit or push if verification fails.
- Do not amend, reset, or discard changes unless explicitly requested.
