## 1. 공통 화면 원칙

- 모바일 360px~430px 폭을 우선 설계한다.
- 데스크톱에서는 중앙 최대 너비 480px의 모바일 앱형 레이아웃으로 표시한다.
- CTA는 화면 하단에 고정하거나 본문 하단에서 명확히 보이게 배치한다.
- 버튼, 선택 카드, 입력창은 터치 영역 높이 44px 이상을 유지한다.
- 의료 진단처럼 보이는 표현은 피하고 `발 유형`, `패턴`, `신호`, `점검` 중심으로 표현한다.
- 모든 화면은 뒤로가기와 새로고침 시 상태 복원이 자연스럽게 동작해야 한다.

### 3.2 Primary Button

| 상태   | 표현                                  |
| ------ | ------------------------------------- |
| 기본   | 진한 색 배경, 흰색 텍스트             |
| 비활성 | 연한 회색 배경, 낮은 대비 텍스트      |
| 로딩   | 텍스트 대신 `처리 중...` 또는 spinner |

### 3.3 Secondary Button

- 이전으로
- 다시 시작하기
- 결과지 확인하기
- 링크형 개인정보 처리방침

- 문항 화면에서만 표시한다.

### 3.5 Toast 또는 Inline Error

- 입력 오류는 해당 입력창 아래 inline으로 표시한다.
- 서버 제출 실패는 CTA 위 또는 화면 하단에 표시한다.

## 4. S-01 인트로

### 4.1 기본 정보

| 항목 | 내용                                              |
| ---- | ------------------------------------------------- |
| 경로 | `/`                                               |
| 목적 | 테스트의 가치를 짧게 전달하고 시작 버튼 클릭 유도 |
| 진입 | 외부 링크, 홈 접근                                |
| 이탈 | `/quiz/1`                                         |

### 4.2 구성

1. 브랜드명: `시즈탱크`
2. 화면 제목: `내 발 유형은 어떤 패턴일까요?`
3. 보조 문구: `8개의 간단한 선택으로 발과 몸의 사용 패턴을 확인해 보세요.`
4. 예상 소요 시간: `약 1분`
5. 시작 CTA: `테스트 시작하기`
6. 보조 링크: `개인정보 처리방침`

### 4.3 동작

- `테스트 시작하기` 클릭 시 `POST /api/session/start`를 호출한다.
- 성공하면 `startToken`, `expiresAt`, 시작 시각을 sessionStorage에 저장한다.
- 기존 답변 상태는 초기화한다.
- `/quiz/1`로 이동한다.
- API 실패 시 `일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.`를 표시한다.

### 4.4 상태

| 상태           | 처리              |
| -------------- | ----------------- |
| 기본           | CTA 활성          |
| 토큰 발급 중   | CTA 로딩          |
| 토큰 발급 실패 | inline error 표시 |

## 5. S-02 문항

### 5.1 기본 정보

| 항목      | 내용                             |
| --------- | -------------------------------- |
| 경로      | `/quiz/[step]`                   |
| step 범위 | 1~8                              |
| 목적      | A/B 답변 수집                    |
| 진입      | `/`, 이전/다음 문항              |
| 이탈      | 다음 문항 또는 `/quiz/analyzing` |

### 5.2 구성

1. 상단 진행 표시: `step / 8`
2. 질문 영역
3. 선택 A 카드
4. 선택 B 카드
5. 하단 이전 버튼
6. 하단 다음 버튼

### 5.3 문항 데이터

| Step | 질문                       | A 라벨                   | B 라벨                     |
| ---- | -------------------------- | ------------------------ | -------------------------- |
| 1    | 평소 나의 서 있는 모습은?  | 양발 균형 / 수직 정렬    | 짝다리 / 한쪽으로 기울어짐 |
| 2    | 걸을 때 나의 발 모양은?    | 발이 정면                | 발이 바깥쪽으로 벌어짐     |
| 3    | 서 있을 때 발의 느낌은?    | 아치 유지 / 3점 지지     | 발 전체가 무너짐           |
| 4    | 나의 옆모습은?             | 골반 중립                | 허리 꺾임 / 배 나옴        |
| 5    | 평소 내 몸통은?            | 갈비뼈 닫힘              | 갈비뼈 벌어짐              |
| 6    | 내 어깨는 편안한가요?      | 어깨가 자연스럽게 내려감 | 어깨 들림 / 목 긴장        |
| 7    | 걸을 때 엉덩이가 쓰이나요? | 발로 밀고 나감           | 발을 던지듯 걷기           |
| 8    | 안면 비대칭이 있나요?      | 균형                     | 한쪽 눈/턱 치우침          |

### 5.4 선택 카드

| 요소        | 설명                              |
| ----------- | --------------------------------- |
| 이미지 영역 | 실제 이미지 없을 경우 placeholder |
| 선택 라벨   | `A` 또는 `B`                      |
| 설명 텍스트 | 문항별 A/B 라벨                   |
| 선택 상태   | 테두리와 배경으로 명확히 표시     |

### 5.5 동작

- A 또는 B 카드를 누르면 해당 답변을 선택한다.
- 선택 즉시 sessionStorage의 `siztank_answers`를 업데이트한다.
- 다음 버튼은 답변이 선택된 경우에만 활성화한다.
- 1번 문항에서 이전 버튼을 누르면 `/`로 이동한다.
- 2~8번 문항에서 이전 버튼을 누르면 이전 step으로 이동한다.
- 1~7번 문항에서 다음 버튼을 누르면 다음 step으로 이동한다.
- 8번 문항에서 다음 버튼을 누르면 결과를 계산하고 `/quiz/analyzing`으로 이동한다.

### 5.6 접근 제한

- `siztank_start_token`이 없으면 `/`로 이동한다.
- 시작 후 30분이 지났으면 상태를 초기화하고 `/`로 이동한다.
- `step`이 1~8 범위가 아니면 `/404`로 이동한다.

## 7. S-04 연락처 입력

### 7.1 기본 정보

| 항목 | 내용                                          |
| ---- | --------------------------------------------- |
| 경로 | `/quiz/contact`                               |
| 목적 | 결과 발송에 필요한 이름, 전화번호와 동의 수집 |
| 진입 | `/quiz/analyzing`                             |
| 이탈 | `/quiz/submitted`                             |

### 7.2 구성

1. 제목: `결과지를 받을 이름과 휴대폰 번호를 입력해 주세요`
2. 보조 문구: `결과 링크를 카카오톡 또는 문자로 보내드립니다.`
3. 이름 입력창
4. 전화번호 입력창
5. 단일 동의 체크박스
6. 개인정보 고지 문구
7. 개인정보 처리방침 링크
8. 제출 CTA: `결과지 받기`
9. 오류 메시지 영역

### 7.3 이름 입력

| 항목        | 내용                            |
| ----------- | ------------------------------- |
| placeholder | `이름`                          |
| 키보드      | 기본 텍스트 키보드              |
| 표시 형식   | 입력값 그대로 표시              |
| 서버 전송   | 원본 입력값 전송. 서버에서 trim |
| 오류        | `이름을 입력해 주세요.`         |

### 7.4 전화번호 입력

| 항목        | 내용                                          |
| ----------- | --------------------------------------------- |
| placeholder | `010-XXXX-XXXX`                               |
| 키보드      | 모바일 numeric keypad                         |
| 표시 형식   | 입력 중 자동 하이픈 허용                      |
| 서버 전송   | 원본 입력값 전송 가능. 서버에서 숫자만 정규화 |
| 오류        | `휴대폰 번호를 다시 확인해 주세요.`           |

### 7.5 체크박스

문구:

```text
시즈탱크 발 유형 결과지와 맞춤 운동 콘텐츠, 혜택 안내를 카카오톡/SMS로 받는 데 동의합니다.
```

### 7.6 개인정보 고지

```text
입력한 이름과 휴대폰 번호는 결과지 발송 및 시즈탱크 콘텐츠·혜택 안내에 사용됩니다. 자세한 내용은 개인정보 처리방침에서 확인할 수 있습니다.
```

### 7.7 CTA 활성 조건

- 이름이 trim 후 1자 이상이다.
- 전화번호가 기본 형식을 만족한다.
- 체크박스가 선택되어 있다.
- 제출 중이 아니다.
- 답변과 결과값이 sessionStorage에 존재한다.
- startToken이 존재한다.

### 7.8 제출 동작

`결과지 받기` 클릭 시 `POST /api/submit`을 호출한다.

요청에 포함할 값:

- `name`
- `phone`
- `consentChecked`
- `consentVersion`
- `startToken`
- `answers`
- `primaryCase`
- `matchedCases`

### 7.9 제출 성공 처리

| 서버 status | 처리                                                             |
| ----------- | ---------------------------------------------------------------- |
| `created`   | `/quiz/submitted?status=created&url={encodedResultUrl}`로 이동   |
| `duplicate` | `/quiz/submitted?status=duplicate&url={encodedResultUrl}`로 이동 |

### 7.10 제출 실패 처리

- 400/429 오류는 서버 message를 inline으로 표시한다.
- 500 오류는 재시도 가능하게 CTA를 다시 활성화한다.
- `TOKEN_EXPIRED` 또는 `INVALID_TOKEN`이면 `처음부터 다시 진행하기` 버튼을 함께 표시한다.

## 8. S-05 제출 완료

### 8.2 Created 상태

#### 구성

1. 제목: `결과지를 보내드렸어요`
2. 보조 문구: `입력한 번호로 카카오톡 또는 문자를 확인해 주세요.`

3. CTA: `기존 결과지 보기`
4. 보조 버튼: `처음부터 다시 하기`

#### 동작

- `기존 결과지 보기` 클릭 시 `url` 파라미터의 결과 URL로 이동한다.
- `처음부터 다시 하기` 클릭 시 sessionStorage를 초기화하고 `/`로 이동한다.

### 8.4 접근 제한

- `status`가 `created` 또는 `duplicate`가 아니면 `/404`로 이동한다.
- `url`이 없거나 내부 결과 URL 형식이 아니면 `/404`로 이동한다.

## 9. S-06 결과지

### 9.1 기본 정보

| 항목 | 내용                                       |
| ---- | ------------------------------------------ |
| 경로 | `/result?primary={caseId}&cases={caseIds}` |
| 목적 | 대표 결과와 함께 나타난 패턴 표시          |
| 진입 | 메시지 링크, 제출 완료 화면                |

### 9.2 URL 예시

```text
/result?primary=case2&cases=case2,case3
```

### 9.3 구성

1. 대표 결과 헤더
2. 내 발 유형 요약
3. 대표 유형 설명
4. 함께 나타난 패턴
5. 주의해야 할 신체 신호
6. 맞춤 운동 영상
7. 추천 시즈탱크 모델
8. 다음 안내

### 9.4 대표 결과 헤더

| 요소       | 설명                 |
| ---------- | -------------------- |
| 라벨       | `대표 발 유형`       |
| 결과명     | 예: `발 기능 붕괴형` |
| 한 줄 요약 | 결과별 headline      |

### 9.5 함께 나타난 패턴

- `cases`에서 `primary`를 제외한 case 목록을 표시한다.
- 추가 case가 없으면 `함께 강하게 나타난 추가 패턴은 없습니다.`를 표시한다.
- `default` 결과에서는 이 섹션을 간결하게 표시하거나 숨길 수 있다.

### 9.6 주의해야 할 신체 신호

- 결과별 `bodySignals` 배열을 리스트로 표시한다.
- 실제 의학적 진단처럼 보이지 않게 `가능성`, `점검`, `신호` 표현을 사용한다.

### 9.7 맞춤 운동 영상

| 상태     | 표시                                   |
| -------- | -------------------------------------- |
| URL 있음 | 영상 썸네일 또는 버튼 `운동 영상 보기` |
| URL 없음 | `맞춤 운동 영상은 준비 중입니다.`      |

### 9.8 추천 시즈탱크 모델

| 상태     | 표시                                |
| -------- | ----------------------------------- |
| URL 있음 | 제품명, 간단 설명, `제품 보기` 버튼 |
| URL 없음 | `추천 제품 정보는 준비 중입니다.`   |

### 9.9 다음 안내

- 카카오톡/SMS를 확인하라는 안내
- 채널 추가 또는 상담 연결은 MVP에서 선택적 placeholder로 둔다.

### 9.10 URL 검증 실패

아래 조건이면 `/404`로 이동한다.

- `primary` 없음
- `cases` 없음
- 허용되지 않은 case ID 포함
- `cases`에 `primary`가 없음
- `default`와 다른 case가 함께 있음

## 14. QA 체크리스트

- 360px 모바일 폭에서 버튼 텍스트가 줄바꿈되어도 깨지지 않는다.
- 8개 문항 모두 A/B 선택이 가능하다.
- 답변 미선택 상태에서는 다음 버튼이 비활성화된다.
- 이전 버튼으로 돌아가면 기존 선택이 유지된다.
- 새로고침 후 답변 상태가 유지된다.
- 30분 만료 상태에서 `/quiz/contact` 접근 시 초기화된다.
- 이름 미입력 오류가 표시된다.
- 전화번호 형식 오류가 표시된다.
- 체크박스 미선택 시 제출할 수 없다.
- 15초 미만 제출은 서버에서 차단된다.
- 중복 번호는 저장 및 발송이 반복되지 않는다.
- 결과지 URL 검증이 동작한다.
- `cases`가 여러 개면 함께 나타난 패턴이 표시된다.
- `default` 결과는 단독으로 표시된다.
- 개인정보 처리방침 링크가 모든 필요한 화면에서 접근 가능하다.
